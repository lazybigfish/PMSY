# 登录接口 500 错误修复记录

## 问题描述

服务器环境更新后，用户无法登录，前端报错：

```
POST http://localhost:3001/auth/v1/token?grant_type=password 500 (Internal Server Error)
登录失败: Error: 邮箱或密码错误
```

## 问题分析

### 根本原因

`api-new/src/services/authService.ts` 中的 `login` 函数在验证失败时抛出的是普通 `Error` 对象：

```typescript
if (!user) {
  throw new Error('邮箱或密码错误');  // ❌ 普通 Error
}
```

但错误处理中间件 `errorHandler` 对普通 `Error` 默认返回 **500 状态码**：

```typescript
// 错误处理中间件逻辑
if (err instanceof AppError) {
  statusCode = err.statusCode;  // 使用自定义状态码
} else {
  statusCode = 500;  // 普通 Error 返回 500
}
```

### 问题影响

1. **HTTP 状态码错误**：登录失败（凭证错误）应该返回 **401 Unauthorized**，但实际返回 **500 Internal Server Error**
2. **前端错误处理混乱**：前端无法正确区分是服务器错误还是用户凭证错误
3. **监控告警干扰**：500 错误会触发服务器告警，而实际上只是用户输入错误

## 修复方案

### 1. 导入正确的错误类型

```typescript
import { UnauthorizedError, ValidationError, NotFoundError, ConflictError } from '../middleware/errorHandler';
```

### 2. 替换 login 函数中的错误类型

```typescript
// 修复前
if (!user) {
  throw new Error('邮箱或密码错误');
}
if (!user.is_active) {
  throw new Error('账号已被禁用');
}
if (!isPasswordValid) {
  throw new Error('邮箱或密码错误');
}

// 修复后
if (!user) {
  throw new UnauthorizedError('邮箱或密码错误');
}
if (!user.is_active) {
  throw new UnauthorizedError('账号已被禁用');
}
if (!isPasswordValid) {
  throw new UnauthorizedError('邮箱或密码错误');
}
```

### 3. 统一替换所有普通 Error

对 `authService.ts` 中所有 `throw new Error` 进行替换：

| 原错误类型 | 替换为 | 适用场景 |
|-----------|--------|---------|
| `new Error('邮箱或密码错误')` | `new UnauthorizedError('邮箱或密码错误')` | 登录验证失败 |
| `new Error('账号已被禁用')` | `new UnauthorizedError('账号已被禁用')` | 账号状态检查 |
| `new Error('旧密码错误')` | `new UnauthorizedError('旧密码错误')` | 密码修改验证 |
| `new Error('用户不存在')` | `new NotFoundError('用户不存在')` | 资源不存在 |
| `new Error('该邮箱已被注册')` | `new ConflictError('该邮箱已被注册')` | 资源冲突 |
| `new Error('该用户名已被使用')` | `new ConflictError('该用户名已被使用')` | 资源冲突 |
| `new Error('该邮箱已被其他用户使用')` | `new ConflictError('该邮箱已被其他用户使用')` | 资源冲突 |
| `new Error('该用户名已被其他用户使用')` | `new ConflictError('该用户名已被其他用户使用')` | 资源冲突 |

## 验证结果

### 修复前

```bash
curl -X POST "http://localhost:3001/auth/v1/token?grant_type=password" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","password":"test123"}'

# 响应
{"error":"Error","code":"INTERNAL_ERROR","message":"邮箱或密码错误","statusCode":500}
```

### 修复后

```bash
curl -X POST "http://localhost:3001/auth/v1/token?grant_type=password" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@test.com","password":"test123"}'

# 响应
{"error":"AppError","code":"UNAUTHORIZED","message":"邮箱或密码错误","statusCode":401}
```

## 经验总结

### 错误处理规范

1. **使用语义化的错误类型**：
   - `UnauthorizedError` (401) - 认证失败、凭证错误
   - `ValidationError` (422) - 参数验证失败
   - `NotFoundError` (404) - 资源不存在
   - `ConflictError` (409) - 资源冲突（重复数据）
   - `BadRequestError` (400) - 请求格式错误

2. **避免直接使用 `new Error`**：
   - 普通 `Error` 会被错误处理中间件识别为 500 服务器错误
   - 应该根据错误场景选择合适的 `AppError` 子类

3. **状态码语义**：
   - 401 Unauthorized - 未授权（凭证错误、Token 过期）
   - 403 Forbidden - 禁止访问（权限不足）
   - 404 Not Found - 资源不存在
   - 409 Conflict - 资源冲突
   - 422 Validation Error - 验证失败
   - 500 Internal Server Error - 服务器内部错误（仅用于真正的服务器错误）
