## 修复记录

### 1. 数据备份页面获取定时备份配置失败修复

**问题描述**：
打开数据备份页面时报错：`获取定时备份配置失败: Unexpected token '<', "<!doctype "... is not valid JSON`

**问题原因**：
`src/services/backupApi.ts` 中的 API 调用使用了原生 `fetch` 直接调用相对路径（如 `/api/system/backup/schedule`），但没有使用 `API_BASE_URL` 前缀。

由于前端运行在 `5173` 端口，请求被发送到前端开发服务器而非后端 API 服务器，返回了 HTML 页面（`index.html`）而非 JSON 数据，导致 JSON 解析错误。

**修复方案**：
将 `backupApi.ts` 中的所有 API 调用从原生 `fetch` 改为使用统一的 `apiClient`，确保使用正确的 `API_BASE_URL`（从环境变量 `VITE_API_URL` 读取，默认为 `http://localhost:3000`）。

**修改文件**：
- `src/services/backupApi.ts`

**主要变更**：
1. 导入方式从 `import { api } from '../lib/api'` 改为 `import { apiClient } from '../lib/api'`
2. 所有 API 方法改用 `apiClient.get/post/put/delete` 统一封装
3. `downloadBackup` 方法因需要返回 `Blob`，仍使用 `fetch` 但显式添加了 `API_BASE_URL` 和认证头

**验证结果**：
- TypeScript 编译通过，无类型错误
- 前端构建成功
- 前端服务已重启（http://localhost:5174/）

### 2. 后端路由顺序问题修复

**问题描述**：
前端 API 调用修复后，后端返回 500 错误：`select * from "backup_records" where "id" = $1 - invalid input syntax for type uuid: "schedule"`

**问题原因**：
Express 路由匹配顺序问题。路由 `/:id` 定义在 `/schedule` 之前，导致请求 `/api/system/backup/schedule` 被错误地匹配到了 `/:id` 路由，将 `schedule` 当作了 `id` 参数。

**修复方案**：
将 `/schedule` 路由（具体路径）移到 `/:id` 路由（参数路径）之前定义。在 Express 中，具体路径的路由应该放在参数路径路由之前，以确保正确匹配。

**修改文件**：
- `api-new/src/routes/system/backup.routes.ts`

**主要变更**：
1. 将 `GET /schedule` 和 `PUT /schedule` 路由定义移动到 `GET /:id` 路由之前
2. 添加注释说明路由顺序的重要性

**验证结果**：
- TypeScript 编译通过
- API 测试成功返回配置数据：`{"config":{"cron":"0 2 * * *","enabled":false,...}}`

### 3. 创建备份功能 500 错误修复

**问题描述**：
点击「创建备份」按钮时，后端返回 500 错误：`column "updated_at" of relation "backup_records" does not exist`

**问题原因**：
`backup_records` 表定义中缺少 `updated_at` 字段，但 `updateById` 函数在更新记录时会自动设置 `updated_at` 字段。

**修复方案**：
1. 修改初始化脚本 `022_create_backup_records.sql`，添加 `updated_at` 字段和触发器
2. 创建迁移脚本 `042_add_backup_records_updated_at.sql`，为现有数据库添加该字段

**修改文件**：
- `api-new/database/init/022_create_backup_records.sql`
- `api-new/database/migrations/042_add_backup_records_updated_at.sql`

**主要变更**：
1. 添加 `updated_at TIMESTAMPTZ DEFAULT NOW() NOT NULL` 字段
2. 创建 `update_backup_records_updated_at` 触发器自动更新该字段
3. 迁移脚本使用 `IF NOT EXISTS` 确保幂等性

**验证结果**：
- 迁移脚本执行成功
- 创建备份 API 测试成功：`{"id":"...","status":"completed","message":"备份任务已创建"}`
- 备份列表 API 正常返回备份数据

### 4. 备份列表日期格式化错误修复

**问题描述**：
打开备份列表页面时报错：`Cannot read properties of undefined (reading 'getTime')`

**问题原因**：
`formatDate` 函数没有处理 `undefined` 或 `null` 的输入。当备份记录的 `createdAt` 字段为空时，调用 `formatDate` 会导致错误。

**修复方案**：
修改 `src/lib/utils.ts` 中的日期格式化函数，添加对 `undefined` 和 `null` 的处理：
1. `formatDate` - 添加空值检查
2. `formatDateTime` - 添加空值检查
3. `formatCreatedAtWithWeek` - 添加空值检查

**修改文件**：
- `src/lib/utils.ts`

**主要变更**：
1. 函数参数类型从 `Date | string` 改为 `Date | string | undefined | null`
2. 添加 `if (!date) return '-';` 空值检查
3. 保持原有格式化逻辑不变

**验证结果**：
- TypeScript 编译通过
- 前端构建成功
- 备份列表页面正常显示

### 5. 备份列表文件大小和创建时间显示问题修复

**问题描述**：
备份创建成功后，备份列表中显示：
- 文件大小为「未知」
- 创建时间为空

**问题原因**：
后端 API 返回的数据使用下划线命名（`file_size`, `created_at`），但前端类型定义和组件期望的是驼峰命名（`fileSize`, `createdAt`），导致数据无法正确显示。

**修复方案**：
在 `BackupService.ts` 中添加 `convertToCamelCase` 函数，将数据库查询结果从下划线命名转换为驼峰命名。

**修改文件**：
- `api-new/src/services/backup/BackupService.ts`

**主要变更**：
1. 添加 `convertToCamelCase` 转换函数
2. 修改 `getBackupList` 函数，对查询结果进行转换
3. 修改 `getBackupById` 函数，对查询结果进行转换

**验证结果**：
- TypeScript 编译通过
- API 测试成功，返回数据格式正确：
  - `"fileSize":257003` - 文件大小正确显示
  - `"createdAt":"2026-03-02T07:23:23.194Z"` - 创建时间正确显示
