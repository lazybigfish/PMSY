# 角色权限更新 500 错误修复记录

## 问题描述

在服务器环境调整角色权限时出现报错：

```
api/rest/v1/app_roles?eq.key=admin:1   Failed to load resource: the server responded with a status of 500 (Internal Server Error)
[API] Update error to app_roles: Error: 请求失败

api/rest/v1/role_permissions?eq.role_key=admin:1   Failed to load resource: net::ERR_CONNECTION_RESET
[API] Delete error from role_permissions: TypeError: Failed to fetch
```

## 问题分析

### 1. 错误日志分析

- **500 错误**：`PATCH api/rest/v1/app_roles?eq.key=admin` - 更新角色时服务器内部错误
- **连接重置**：`DELETE api/rest/v1/role_permissions?eq.role_key=admin` - 删除权限时连接被重置

### 2. 代码审查

通过检查后端代码，发现 `dbService.ts` 中的 `update` 和 `updateById` 方法会自动添加 `updated_at` 字段：

```typescript
// dbService.ts L240, L248, L274, L282
.update({ ...data, updated_at: new Date() })
```

但检查数据库表结构发现：

**`app_roles` 表定义**（`010_create_app_roles.sql`）：
```sql
CREATE TABLE IF NOT EXISTS app_roles (
    key TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    description TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL
    -- 缺少 updated_at 字段！
);
```

**`role_permissions` 表定义**（`010_create_app_roles.sql`）：
```sql
CREATE TABLE IF NOT EXISTS role_permissions (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    role_key TEXT REFERENCES app_roles(key) ON DELETE CASCADE,
    module_key TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL
    -- 缺少 updated_at 字段！
);
```

### 3. 根本原因

当后端尝试更新 `app_roles` 或 `role_permissions` 表时，`dbService` 会自动添加 `updated_at: new Date()` 到更新数据中，但这两个表在创建时没有定义 `updated_at` 字段，导致 PostgreSQL 报错：

```
column "updated_at" of relation "app_roles" does not exist
```

这个数据库错误被后端捕获后返回 500 内部服务器错误。

## 修复方案

### 方案选择

有两种修复方案：

1. **修改 dbService.ts**：在更新前检查表是否有 `updated_at` 字段
2. **修改数据库表结构**：给 `app_roles` 和 `role_permissions` 表添加 `updated_at` 字段

选择 **方案 2**，因为：
- 更简单直接
- 符合数据库设计规范（所有表都应该有 `created_at` 和 `updated_at`）
- 与其他表的命名规范保持一致

### 修复内容

#### 1. 修改初始化 SQL（`010_create_app_roles.sql`）

```sql
-- 创建 app_roles 表（应用角色）
CREATE TABLE IF NOT EXISTS app_roles (
    key TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    description TEXT,
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL,
    updated_at TIMESTAMPTZ DEFAULT NOW() NOT NULL  -- 新增
);

-- 创建 role_permissions 表（角色权限）
CREATE TABLE IF NOT EXISTS role_permissions (
    id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
    role_key TEXT REFERENCES app_roles(key) ON DELETE CASCADE,
    module_key TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT NOW() NOT NULL,
    updated_at TIMESTAMPTZ DEFAULT NOW() NOT NULL,  -- 新增
    UNIQUE(role_key, module_key)
);
```

#### 2. 创建数据库迁移文件（`043_add_app_roles_updated_at.sql`）

```sql
-- 为 app_roles 和 role_permissions 表添加 updated_at 字段
-- 修复角色权限更新时的 500 错误

-- 为 app_roles 表添加 updated_at 字段
ALTER TABLE app_roles
ADD COLUMN IF NOT EXISTS updated_at TIMESTAMPTZ DEFAULT NOW() NOT NULL;

-- 为 role_permissions 表添加 updated_at 字段
ALTER TABLE role_permissions
ADD COLUMN IF NOT EXISTS updated_at TIMESTAMPTZ DEFAULT NOW() NOT NULL;

-- 更新现有数据的 updated_at 字段为 created_at 的值
UPDATE app_roles SET updated_at = created_at WHERE updated_at IS NULL;
UPDATE role_permissions SET updated_at = created_at WHERE updated_at IS NULL;
```

## 修复步骤

### 问题1：500 错误修复

**原因**：`app_roles` 和 `role_permissions` 表缺少 `updated_at` 字段

**修复**：
1. 已更新 `010_create_app_roles.sql` - 新部署将自动包含 `updated_at` 字段
2. 已创建迁移文件 `043_add_app_roles_updated_at.sql` - 现有环境升级时使用
3. 已在服务器执行迁移，添加字段成功

### 问题2：DELETE 请求连接重置修复

**原因**：云服务商（阿里云/腾讯云等）的安全防护或防火墙拦截了 DELETE 请求

**修复方案**：使用 POST 请求替代 DELETE

1. **后端新增端点**（`api-new/src/routes/rest.ts`）：
```typescript
// POST /rest/v1/role_permissions/save
// 使用事务：先删除旧权限，再插入新权限
router.post('/role_permissions/save', requireAuth, async (req, res) => {
  const { role_key, module_keys } = req.body;
  await db.transaction(async (trx) => {
    await trx('role_permissions').where('role_key', role_key).delete();
    if (module_keys.length > 0) {
      await trx('role_permissions').insert(
        module_keys.map((moduleKey) => ({ role_key, module_key: moduleKey }))
      );
    }
  });
  res.json({ success: true });
});
```

2. **前端修改**（`src/services/roleService.ts`）：
```typescript
export async function saveRolePermissions(roleKey: string, moduleKeys: string[]): Promise<void> {
  const response = await fetch(
    `${import.meta.env.VITE_API_URL}/rest/v1/role_permissions/save`,
    {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${localStorage.getItem('access_token')}`,
      },
      body: JSON.stringify({ role_key: roleKey, module_keys: moduleKeys }),
    }
  );
  if (!response.ok) throw new Error('保存权限失败');
}
```

3. **部署完成**：
- 后端 API 已重新构建并部署
- 前端代码已构建并部署
- 新端点测试通过（返回 401 表示端点正常工作，只是需要有效 token）

## 经验教训

1. **数据库设计规范**：所有表都应该包含 `created_at` 和 `updated_at` 字段，这是基本的时间戳追踪规范
2. **代码与 Schema 同步**：当代码自动添加某些字段时，必须确保数据库 Schema 支持这些字段
3. **错误日志分析**：500 错误通常意味着服务端代码执行出错，需要检查后端日志定位具体问题
4. **迁移脚本重要性**：对于已有数据的表结构变更，必须通过迁移脚本执行，确保数据完整性
