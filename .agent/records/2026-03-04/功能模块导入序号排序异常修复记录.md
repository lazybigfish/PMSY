# 功能模块导入序号排序异常修复记录

## 问题描述

服务器环境功能模块页面导入模块后，序号显示异常：
- **预期显示**：1, 2, 3, 4, 5, 6, 7, 8, 9...
- **实际显示**：1, 2, 24, 42, 43, 44, 45, 46, 56...

![问题截图](用户提供的截图显示序号不连续)

## 问题定位

**文件位置**：`src/pages/projects/tabs/FunctionalModules.tsx`

**问题函数**：`processImport`（第493行开始）

**问题代码**（第536-541行）：
```typescript
// 计算当前模块的 path
const siblingIndex = parentStack.length > 0 
  ? parentStack.filter(p => p.level === level - 1).length - 1
  : i;
const path = parentPath 
  ? `${parentPath}.${siblingIndex + 1}`
  : String(siblingIndex + 1);
```

## 问题原因分析

### 根本原因
`siblingIndex` 的计算逻辑错误：
- 原逻辑使用 `parentStack.filter(p => p.level === level - 1).length - 1` 来统计父级栈中层级为 `level-1` 的元素数量
- 这实际上统计的是**父级**的数量，而不是**当前层级**的兄弟节点数量
- 导致同级索引计算错误，path 生成不正确

### 举例说明
假设导入以下模块结构：
```
层级1: 办公用房管理子系统
层级1: 大数据融合中心
层级1: 公物仓子系统  ← 应该是 path=3，但计算错误
```

原逻辑中：
- 当处理第3个模块时，`parentStack` 中可能包含前面处理过的模块
- `filter(p => p.level === 0)` 返回空数组（因为没有 level 为 0 的元素）
- 结果 `siblingIndex = -1`，导致 path 计算错误

## 修复方案

### 修复思路
引入一个按层级统计兄弟节点数量的计数器 `levelSiblingCount`，独立维护每个层级的兄弟节点索引。

### 修复代码
```typescript
// 按层级统计兄弟节点数量，用于生成正确的 path
const levelSiblingCount: Record<number, number> = {};

// ... 在循环中 ...

// 初始化当前层级的兄弟计数
if (levelSiblingCount[level] === undefined) {
  levelSiblingCount[level] = 0;
}
// 当前层级的兄弟索引
const siblingIndex = levelSiblingCount[level];
// 递增当前层级的兄弟计数
levelSiblingCount[level]++;

const path = parentPath 
  ? `${parentPath}.${siblingIndex + 1}`
  : String(siblingIndex + 1);
```

### 修复后逻辑
- 每个层级（level）独立维护自己的兄弟节点计数器
- 导入第1个 level=1 的模块：`levelSiblingCount[1] = 0`，path = "1"，然后计数器变为1
- 导入第2个 level=1 的模块：`levelSiblingCount[1] = 1`，path = "2"，然后计数器变为2
- 导入第3个 level=1 的模块：`levelSiblingCount[1] = 2`，path = "3"，然后计数器变为3
- 以此类推，确保序号连续正确

## 修复验证

### 测试场景
1. 导入平级模块（全部 level=1）
2. 导入多级模块（level=1,2,3 混合）
3. 导入复杂层级结构

### 预期结果
- 序号显示连续：1, 2, 3, 4, 5...
- 多级模块 path 正确：1, 2, 2.1, 2.2, 3...
