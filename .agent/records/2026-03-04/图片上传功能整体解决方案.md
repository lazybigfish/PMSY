## 二、存在的问题

### 2.1 架构层面的问题

#### 问题1：多重上传路径混乱

**现状**：系统中有**3种不同的上传方式**

```typescript
// 方式1: fileService.uploadFile (XMLHttpRequest)
// 用于: PostImageUploader, FileManager
await fileService.uploadFile(file, folderId, onProgress);

// 方式2: api.storage.from().upload (Fetch API)
// 用于: TaskProgressUpdateModal, MilestoneTaskList
await api.storage.from('bucket').upload(path, file);

// 方式3: 直接XMLHttpRequest
// 用于: PostImageUploader内部
const xhr = new XMLHttpRequest();
xhr.open('POST', `${API_BASE_URL}/storage/v1/object/files/forum`);
```

**问题**：
- 维护困难，修改上传逻辑需要改多处
- 行为不一致（进度回调、错误处理不同）
- 文件名生成逻辑分散

#### 问题2：文件名生成逻辑不一致

**后端 storage.ts (第115行)**：
```typescript
// 使用前端提供的文件路径，不再重新生成文件名
const fullPath = filePath;
```

**后端 storageService.ts (第95-101行)**：
```typescript
export function generateUniqueFilename(originalName: string): string {
  const ext = path.extname(originalName);
  const basename = path.basename(originalName, ext);
  const timestamp = Date.now();
  const uniqueId = uuidv4().split('-')[0];
  return `${basename}_${timestamp}_${uniqueId}${ext}`;
}
```

**问题**：
- 前端各自生成文件名，格式不统一
- 有些场景重复生成文件名导致404
- 没有统一的路径规范

#### 问题3：URL生成和访问方式混乱

**方式1 - fileService.getFileUrl**：
```typescript
export function getFileUrl(filePath: string): string {
  const fileName = filePath.split('/').pop() || filePath;
  return `${API_BASE_URL}/storage/v1/object/public/files/forum/${fileName}`;
}
```

**方式2 - api.storage.from().getPublicUrl**：
```typescript
getPublicUrl: (path: string): { data: { publicUrl: string } } => ({
  data: {
    publicUrl: `${API_BASE_URL}/storage/v1/object/public/${bucket}/${path}`
  }
});
```

**方式3 - storageService.getPublicUrl**：
```typescript
export function getPublicUrl(bucket: string, filePath: string): string {
  return `${API_URL}/storage/v1/object/public/${bucket}/${filePath}`;
}
```

**问题**：
- URL格式不统一
- 有些场景文件找不到（路径拼接错误）
- HTTPS混合内容问题

#### 问题4：错误处理不完善

**现状**：
- 上传失败没有统一的重试机制
- 错误信息不友好（直接暴露技术错误）
- 没有断点续传或分片上传

#### 问题5：数据库记录与文件存储分离

**现状**：
- 文件上传到MinIO后，数据库记录通过额外API调用创建
- 存在文件上传成功但数据库记录失败的风险
- 没有事务保证一致性

## 三、根本原因总结

### 3.1 设计层面

1. **缺乏统一的上传抽象层**：各组件各自实现上传逻辑
2. **前后端契约不清晰**：文件名生成、路径格式没有统一定义
3. **存储服务设计过度复杂**：兼容Supabase API但自身需求更简单

### 3.2 实现层面

1. **代码重复**：相同功能在多处实现
2. **错误处理不一致**：有的用try-catch，有的用error回调
3. **类型定义分散**：FileRecord定义在types/file.ts，但上传返回类型不一致

### 4.2 核心设计原则

1. **单一入口**：所有上传通过统一的 `UploadManager`
2. **前后端契约明确**：统一定义请求/响应格式
3. **原子操作**：文件存储和数据库记录在一个事务中完成
4. **渐进增强**：支持从简单上传到分片上传的升级

#### 5.2.3 数据库事务保证

```typescript
// 使用数据库事务确保原子性
async uploadWithRecord(file: Buffer, options: UploadOptions): Promise<FileRecord> {
  return await db.transaction(async (trx) => {
    // 1. 上传文件到MinIO
    const storageResult = await this.storage.upload(file, options);
    
    // 2. 创建数据库记录
    const fileRecord = await trx('files').insert({
      name: options.fileName,
      storage_path: storageResult.path,
      storage_type: 'minio',
      size: file.length,
      mime_type: options.mimeType,
      url: storageResult.url,
      // ... 其他字段
    }).returning('*');
    
    // 3. 如果有上下文关联，创建关联记录
    if (options.metadata?.taskId) {
      await trx('task_attachments').insert({
        task_id: options.metadata.taskId,
        file_id: fileRecord.id,
        // ...
      });
    }
    
    return fileRecord;
  });
}
```

### 6.1 文件名规范

```typescript
// 统一文件名生成规则
function generateFileName(originalName: string, options?: { prefix?: string }): string {
  const timestamp = Date.now();
  const random = Math.random().toString(36).substring(2, 8);
  const ext = path.extname(originalName);
  const base = path.basename(originalName, ext);
  
  // 格式: {prefix}/{timestamp}_{random}_{basename}.{ext}
  // 示例: forum/1677830400000_a1b2c3_image.jpg
  const fileName = `${timestamp}_${random}_${base}${ext}`;
  
  return options?.prefix ? `${options.prefix}/${fileName}` : fileName;
}
```
